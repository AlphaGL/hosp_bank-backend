Authentication

All endpoints require a Bearer JWT token in the Authorization header, except /api/auth/login/. Tokens expire after 8 hours; refresh tokens last 7 days.

🔑 Auth Flow
1
Login
POST to /api/auth/login/ with username + password. Receive access and refresh tokens.
2
Use Token
Add Authorization: Bearer <access> header to all requests.
3
Refresh
When access token expires, POST to /api/auth/refresh/ with the refresh token.
POST /api/auth/login/ PUBLIC Authenticate and receive JWT tokens ▶
Request Body
usernamestringreqStaff username
passwordstringreqStaff password
Response 200
{
  "access": "eyJhbGci...",
  "refresh": "eyJhbGci...",
  "user": {
    "id": 1,
    "username": "admin",
    "role": "admin",
    "department": "Management"
  }
}
POST /api/auth/logout/ JWT Blacklist refresh token (logout) ▶
Request Body
refreshstringreqThe refresh token to invalidate
Response 200
{ "message": "Logged out successfully." }
POST /api/auth/refresh/ PUBLIC Get a new access token ▶
Request Body
refreshstringreqValid refresh token
Response 200
{ "access": "eyJhbGci..." }
GET /api/auth/me/ JWT Get logged-in user's profile ▶
Response 200 — Staff Object
{
  "id": 1, "username": "jane", "first_name": "Jane", "last_name": "Doe",
  "full_name": "Jane Doe", "email": "jane@hospital.ng",

💡 Also supports PATCH to update profile fields (first_name, last_name, phone, email, department).

Roles & Permissions

Every staff user has a role field that controls access. Admins can access everything.

Role Value Display Name Key Permissions
receptionist Receptionist Register patients, create visits, view queue & notifications
finance Finance Staff Create & confirm payments, view billing records
doctor Doctor / Technician View & manage queue (own dept), update service status, upload reports. Auto-filtered to their department.
admin Admin Full access to all endpoints including staff management, service catalogue CRUD

Patients

Base: /api/patients/records/

Manage patient records. All authenticated staff can view; any authenticated user can register a patient (auto-assigns registered_by).

Query Params: ?search= (patient_id, name, phone)  |  ?ordering=created_at

GET /api/patients/records/ JWT List all patients (paginated) ▶
Response — Paginated List

POST /api/patients/records/ JWT Register a new patient ▶
Request Body

last_namestringreqPatient last name
ageintegerreqPatient age

Send only the fields you want to update. All fields optional in PATCH.

Base: /api/patients/visits/

A Visit represents one hospital encounter for a patient. When created, status is auto-set to awaiting_payment.

Query Params: ?status=  |  ?priority=normal|emergency  |  ?date=YYYY-MM-DD

GET /api/patients/visits/ JWT List visits (filterable) ▶
Visit Object Fields

patient_id_codestringreade.g. PAT-00001

Request Body
patientintegerreqPatient DB ID
prioritystringopt"normal" (default) or "emergency"
notesstringoptClinical notes
Auto-set Fields
✅ status → awaiting_payment
✅ created_by → current user

Full CRUD on staff users. Admin-only. Use ?search= to filter by username, name, email, or role.

GET /api/patients/staff/ ADMIN List all staff ▶
Returns list of Staff objects ordered by first_name. Supports ?search= on username, first_name, last_name, email, role.

Visit Services

Base: /api/services/visit-services/

Services booked for a specific visit. When created, price_at_booking is snapshotted from the catalogue. Doctors only see their own department's services.

Query Params: ?visit=  |  ?department=  |  ?status=

POST /api/services/visit-services/ JWT Attach a service to a visit ▶
Request Body
visitintegerreqVisit ID
serviceintegerreqDiagnosticService ID
notesstringoptOptional clinical note
Auto-set
✅ price_at_booking is copied from service's base_price at creation time — price changes won't affect existing bookings.
PATCH /api/services/visit-services/{id}/update-status/ JWT Update service status ▶
Request Body
statusstringreqpending | in_progress | completed | cancelled
Side Effects
🔁 in_progress auto-sets started_at.
✅ completed auto-sets completed_at.
🏁 If ALL services complete → Visit status → completed.
POST /api/services/visit-services/{id}/upload-report/ JWT Upload a report file (PDF/image) ▶
Request (multipart/form-data)
reportfilereqPDF or image file. Use Content-Type: multipart/form-data
Response 200
{
  "message": "Report uploaded.",
  "report_url": "https://.../media/reports/..."
}

Billing / Payments

Base: /api/billing/payments/

One Payment record per Visit. Finance staff create the record, then confirm it. Confirming a payment triggers queue placement and notifications.

Query Params: ?status=pending|paid|failed|refunded  |  ?date=YYYY-MM-DD  |  ?patient=PAT-00001

POST /api/billing/payments/ FINANCE Create payment record for a visit ▶
Request Body
visitintegerreqVisit ID (1:1 relation)
Auto-set
✅ amount_due is auto-calculated from all visit services' price_at_booking values.
✅ receipt_number auto-generated (RCPxxxxxxxx).
✅ status → pending.
GET /api/billing/payments/ JWT List all payments ▶
Payment Object Fields
receipt_numberstringreadUnique receipt ID e.g. RCPA1B2C3D4
patient_namestringreadFull patient name
amount_duedecimalreadTotal bill amount (₦)
amount_paiddecimalreadAmount actually paid
balancedecimalreadamount_due - amount_paid
is_fully_paidbooleanreadTrue when amount_paid ≥ amount_due
statusstringreadpending | paid | failed | refunded
services_summaryarrayreadList of {service, price} objects
POST /api/billing/payments/{id}/confirm/ FINANCE Confirm payment received ⚡ Key endpoint ▶
Request Body
amount_paiddecimalreqAmount received from patient
payment_methodstringreqcash | card | bank_transfer
transaction_refstringoptBank/card ref number
notesstringoptAny billing notes
Chain Reaction (if fully paid)
🚀 When amount_paid ≥ amount_due:

1. Payment status → paid
2. Visit status → paid
3. Queue entries created per department
4. Visit status → in_queue
5. Notifications sent to receptionists & admin
GET /api/billing/payments/{id}/receipt/ JWT Get receipt data (paid only) ▶
⚠️ Returns 400 if payment is not yet in paid status. Includes a receipt_generated_at timestamp.

Department Queues

Base: /api/queues/

Auto-managed queue entries per department per day. Created automatically when payment is confirmed. Emergency patients are bumped to position #1. Doctors auto-see only their department.

Query Params: ?department=  |  ?status=waiting|in_progress|completed|skipped  |  ?date=YYYY-MM-DD (default: today)  |  ?priority=emergency

GET /api/queues/live/?department={name} JWT Live queue — today's waiting patients ▶
Response — ordered by: emergency first, then queue_number
[{
  "id": 5, "queue_number": 1,
  "patient_name": "Emeka Okafor",
  "patient_id_code": "PAT-00001",
  "priority": "emergency",
  "department": "Radiology",
  "status": "waiting",
  "services": ["Chest X-Ray"]
}]
PATCH /api/queues/{id}/call/ JWT Call patient to room (in_progress) ▶
No request body needed. Sets status → in_progress, records called_at and attended_by (current user).
PATCH /api/queues/{id}/complete/ JWT Mark queue entry as completed ▶
Sets status → completed and records completed_at.
PATCH /api/queues/{id}/skip/ JWT Skip a patient (no-show) ▶
Sets status → skipped. Patient stays in queue record for reporting.
GET /api/queues/summary/ JWT Department-wise count summary for today ▶ 
{
  "Radiology": { "waiting": 4, "in_progress": 1, "completed": 12 },
  "Ultrasound": { "waiting": 2, "completed": 8 }
}

Auto-generated notifications for system events. Each user sees only notifications targeted at their role. Read-only — notifications are created by the system, not the frontend.

Patient Journey Workflow

The complete step-by-step order of API calls for a patient visit — from registration to completion.

📍 Step-by-Step Flow
1
Receptionist: Register Patient
POST /api/patients/records/
Creates patient record, returns patient.id for next step
2
Receptionist: Create Visit
POST /api/patients/visits/ { patient: id, priority: "normal" }
Visit status → awaiting_payment
3
Receptionist: Attach Services
POST /api/services/visit-services/ { visit: id, service: id }
Repeat for each requested test. Price snapshotted automatically.
4
Finance: Create Payment Record
POST /api/billing/payments/ { visit: id }
Amount auto-totalled from services. Patient shown the bill.
5
Finance: Confirm Payment ⚡
POST /api/billing/payments/{id}/confirm/
This single call: marks paid → places in queue per department → sends notifications
6
Doctor: View Queue & Call Patient
GET /api/queues/live/ then PATCH /api/queues/{id}/call/
Doctor sees their dept queue, calls the next patient
7
Doctor: Complete Service & Upload Report
PATCH /api/services/visit-services/{id}/update-status/ then POST .../upload-report/
When all services complete → visit auto-marked completed → receptionist notified
8
Finance: Issue Receipt
GET /api/billing/payments/{id}/receipt/
Patient discharged with printed receipt

Error Reference

HTTP Status Codes
200OK — request succeeded
201Created — resource created successfully
204No Content — successful DELETE
400Bad Request — validation error, check error body for field-level details
401Unauthorized — missing or invalid JWT token
403Forbidden — authenticated but insufficient role (e.g. receptionist hitting finance endpoint)
404Not Found — resource does not exist
500Server Error — unexpected server-side issue
Validation Error Format (400)
{
  "field_name": ["This field is required."],
  "another_field": ["Ensure this value has at most 20 characters."]
}
Auth Error Format (401)
{ "error": "Invalid credentials." }
Pagination Format
{
  "count": 150,
  "next": "https://api.../endpoint/?page=2",
  "previous": null,
  "results": [ /* items */ ]
}
📄 All list endpoints are paginated at 20 items per page. Use ?page=N to paginate.
